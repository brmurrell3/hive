# Hive Implementation Guide

This document contains every architectural decision, API contract, data format, and design constraint needed to implement Hive's four-phase roadmap. It is the single source of truth for implementation. Read the companion documents for business context (hive-product-vision.md) and requirement summaries (hive-technical-requirements.md).

**Audience:** An implementer (human or AI) who has access to the codebase and needs to make zero design decisions -- only implementation decisions.

### Existing Sidecar HTTP API

The sidecar listens on `:9100` (configurable via `Config.HTTPAddr`). Routes are registered in `internal/sidecar/health.go:setupHTTPRoutes()`.

| Method | Path | What it does |
|---|---|---|
| GET | /health | Returns agent health status, uptime, runtime state. Unauthenticated. |
| GET | /capabilities | Returns list of declared capabilities for this agent. |
| POST | /capabilities/{name}/invoke | Invokes a capability. Executes locally via `router.CallLocal()`. Request body: `{"inputs": {...}}`. Response: `{"status": "success", "outputs": {...}}` or `{"status": "error", "error": {...}}`. |

**Authentication:** Bearer token via `Authorization` header. Token is auto-generated by hived and passed to the sidecar via config. The /health endpoint is exempt.

**Missing endpoints needed for Phase 1-2:**

| Method | Path | Purpose | Phase |
|---|---|---|---|
| POST | /capabilities/{name}/invoke-remote | Invoke a capability on a *different* agent via NATS. Uses `router.Invoke(targetAgentID, ...)`. | Phase 1 |
| GET | /team/capabilities | List all capabilities available from team members (discovered via NATS registry). | Phase 2 |

The invoke-remote endpoint is needed so that the lead agent's orchestration script can call peer agents' capabilities through HTTP. The local invoke endpoint only calls handlers registered on the same agent.

### NATS Subject Map

All subjects use the `hive.` prefix. Format strings are in `internal/protocol/subjects.go`.

| Subject | Direction | Purpose |
|---|---|---|
| `hive.health.{AGENT_ID}` | Agent -> Control plane | Heartbeat every 30s (configurable). |
| `hive.control.{AGENT_ID}` | Control plane -> Agent | Control commands (stop, restart, reload). |
| `hive.capabilities.register` | Agent -> Control plane | Agent publishes its capability list on startup. |
| `hive.capabilities.{AGENT_ID}.{CAP}.request` | Agent -> Agent | Capability invocation request. |
| `hive.team.{TEAM_ID}.broadcast` | Lead -> Team members | Team-wide broadcast from lead agent. |
| `hive.agent.{AGENT_ID}.inbox` | Any -> Agent | Direct messaging. |
| `hive.agent.{AGENT_ID}.memory` | Control plane -> Agent | MEMORY.md hot-reload push. |

### Message Envelope

Every NATS message uses this JSON envelope (defined in `internal/types/envelope.go`):

```
{
  "id":             "uuid-v4",              // required
  "from":           "agent-id",             // required
  "to":             "target-agent-or-team", // required
  "type":           "capability-request",   // required, one of: task, result, broadcast, status, health, control, capability-request, capability-response, error, memory-update, node-heartbeat
  "timestamp":      "2026-03-10T12:00:00Z", // required, RFC3339, must be within 5 minutes of now
  "payload":        { ... },                // required, JSON object (max 2MB)
  "reply_to":       "nats-inbox-subject",   // optional
  "correlation_id": "trace-id"              // optional, max 256 chars
}
```

### Capability Invocation Protocol

When Agent A invokes Agent B's capability:

1. Agent A calls `router.Invoke("agent-b", "review-code", inputs, 30s)`
2. Router publishes to `hive.capabilities.agent-b.review-code.request` with a NATS reply subject
3. Agent B's router receives the message, dispatches to the registered handler
4. Handler returns `(outputs, error)`
5. Router publishes response to the reply subject
6. Agent A receives the response (or timeout after 30s + 5s NATS buffer)

Response format:

```
{
  "capability": "review-code",
  "status":     "success" | "error" | "timeout",
  "outputs":    { ... },        // when status=success
  "error":      { "code": "...", "message": "...", "retryable": bool },  // when status=error
  "duration_ms": 1234
}
```

Error codes: AGENT_OFFLINE, CAPABILITY_NOT_FOUND, CAPABILITY_TIMEOUT, CAPABILITY_FAILED, INVALID_INPUTS, UNAUTHORIZED, PAYLOAD_TOO_LARGE, INTERNAL_ERROR.

Circuit breaker: per-agent, per-capability. Trips after 5 consecutive errors in 60s. Open for 30s, then half-open (one test request). Configured in `internal/capability/router.go`.

## 2. Phase 1: The Demo

### 2.1 Agent Architecture

All three demo agents follow the same pattern: a shell script (or small Go/Python program) that communicates with the sidecar via HTTP on localhost:9100.

**Agent process lifecycle:**
1. hived starts the agent process via the process backend (`internal/backend/process/backend.go`)
2. The process receives environment variables: `HIVE_AGENT_ID`, `HIVE_TEAM_ID`, `HIVE_NATS_URL`, `HIVE_NATS_TOKEN`, `HIVE_SIDECAR_URL` (default `http://localhost:9100`)
3. The sidecar is started by hived in-process (library mode) alongside the agent process, not as a separate binary
4. The agent process does its work (waits for invocations, processes requests, returns results)
5. Capability invocations arrive via the sidecar's registered handlers -- the sidecar calls the agent process via a local HTTP callback or stdin/stdout protocol

**Decision: Agent-to-sidecar communication for capability handling.**
The sidecar registers capability handlers at startup (from the manifest). When a remote invocation arrives via NATS, the sidecar needs to forward it to the agent process. Two options exist in the codebase:

- **Option A (current):** The sidecar's `executeCapabilityLocally()` method forwards to a configurable HTTP endpoint on the agent process (e.g., `http://localhost:9200/handle/{capability}`). The agent process runs its own HTTP server.
- **Option B:** The agent process polls the sidecar via `GET /messages/receive` (documented in `docs/communication.md` but not yet implemented as an HTTP endpoint).

**Design decision for Phase 1:** Use Option A. Each demo agent runs a minimal HTTP server on a port specified in the manifest. The sidecar forwards capability requests to this endpoint. This is the simplest path that works with any language runtime.

The agent manifest specifies the callback port via environment variable. The sidecar reads `HIVE_AGENT_CALLBACK_URL` to know where to forward invocations. Default: `http://localhost:920{N}` where N is derived from agent startup order (or explicitly configured in manifest).

### 2.3 Pipeline Orchestration

The lead agent (code-reviewer) is responsible for orchestrating the pipeline. When triggered, it:

1. Receives a trigger via team broadcast (`hive.team.ci-pipeline.broadcast`)
2. Invokes `test-runner.run-tests` and `security-scanner.scan-security` capabilities in parallel via the sidecar's invoke-remote endpoint
3. Waits for both responses (30s timeout each)
4. Aggregates results into a pipeline report
5. Publishes the report to stdout and to the team results subject

**Trigger mechanism:** For Phase 1, the trigger is a CLI command:

```
hivectl trigger --cluster-root ./demo --team ci-pipeline --payload '{"repo_path": "/path/to/repo", "test_command": "go test ./..."}'
```

This publishes a broadcast message to `hive.team.ci-pipeline.broadcast`. The lead agent receives it and starts the pipeline.

`hivectl trigger` is a new subcommand. It connects to the embedded NATS and publishes a single envelope with type `task` to the team broadcast subject.

### 2.5 hivectl dev

New subcommand in `cmd/hivectl/`. Behavior:

1. Run `config.LoadAndValidate(clusterRoot)` -- fail fast on invalid manifests
2. Set environment: `HIVE_TEST_FIRECRACKER=mock` (force process backend)
3. Start hived in-process (not as a subprocess) with:
   - Log level: debug
   - Log format: text (not JSON)
   - Cluster root: the provided path
4. Block until SIGINT/SIGTERM
5. On manifest file change (detected by the existing watcher), log which agents are restarting

**Alternative implementation:** `hivectl dev` could simply exec `hived` with the right flags. This is simpler and keeps hivectl as a thin client. The flags needed: `hived --cluster-root PATH --log-level debug --log-format text --force-process-backend`. The `--force-process-backend` flag is new and tells hived to use the process backend for all agents regardless of `spec.tier`.

**Design decision:** Use the exec approach. Simpler, less code, and hivectl stays a CLI client that doesn't embed server logic.

## 3. Phase 2: Developer Experience

### 3.1 OpenClaw Runtime

When `runtime.type: openclaw` is set in a manifest, the backend must:

1. **Locate the OpenClaw binary.** Check `PATH` for `openclaw` or `node` + the OpenClaw dist directory. If not found, return a clear error message with install instructions.

2. **Generate workspace.** Copy the agent directory's OpenClaw files (SOUL.md, USER.md, AGENTS.md, IDENTITY.md, skills/) into the agent's workspace directory at `.state/agents/{AGENT_ID}/workspace/`.

3. **Generate openclaw.json** from the manifest. Minimum viable config:
```json
{
  "gateway": { "bind": "localhost", "port": 18789 },
  "agents": {
    "defaults": {
      "model": "{spec.runtime.model.provider}/{spec.runtime.model.name}"
    }
  }
}
```

4. **Start the OpenClaw process** with the workspace as `~/.openclaw/workspace` (via environment variable or symlink).

5. **Bridge capabilities.** The sidecar translates Hive capability invocations into OpenClaw HTTP API calls (OpenClaw's gateway runs on port 18789 inside the agent's network namespace). The sidecar's `executeCapabilityLocally()` method sends the invocation to OpenClaw's gateway as a chat message or skill invocation and parses the response.

**Constraint:** OpenClaw's gateway assumes it owns the network port. On the process backend, each OpenClaw agent needs its own port. Use the agent's configured callback port or auto-assign from a range.

### 3.2 Agent SDK Contract

The SDK is a convenience wrapper. The underlying contract is HTTP against the sidecar.

**Startup sequence:**
1. Read environment: `HIVE_AGENT_ID`, `HIVE_TEAM_ID`, `HIVE_SIDECAR_URL`, `HIVE_CALLBACK_PORT`
2. Start an HTTP server on `HIVE_CALLBACK_PORT`
3. Register capability handlers on routes: `POST /handle/{capability_name}`
4. Block, responding to invocations and health checks

**Capability handler contract:**
- Request: `POST /handle/{capability_name}` with body `{"inputs": {"key": "value", ...}}`
- Response: `{"outputs": {"key": "value", ...}}` with status 200, or `{"error": {"code": "...", "message": "..."}}` with status 500

**Health check contract:**
- The sidecar handles heartbeats. The agent process does not need to implement health checks.
- If the agent process exits, the sidecar detects it and reports unhealthy.

**SDK deliverables:**
- Python: `hive_sdk` package. Single file. No dependencies beyond `http.server` and `json`.
- Go: `sdk/hive` package. Already in the repo structure at `sdk/`.
- TypeScript: `@hive/sdk` package. Single file. No dependencies beyond `http` and `node:http`.

### 3.3 Cross-Team Capability Discovery

Current state: capabilities are registered to `hive.capabilities.register` on startup. The control plane stores them in SQLite (`internal/state/store.go`). `hivectl capabilities list` displays them.

**What's needed:** The sidecar needs to query the control plane for capabilities outside its own team. This requires a new NATS request-reply pattern:

- Agent sidecar publishes request to `hive.ctl.capabilities.list` (already exists in protocol/subjects.go)
- Control plane responds with full capability registry
- Sidecar filters for relevant capabilities and exposes them via `GET /team/capabilities` HTTP endpoint

For invocation, the sidecar uses `router.Invoke(targetAgentID, capName, inputs, timeout)` which already works cross-team -- the NATS subject is agent-scoped, not team-scoped.

**RBAC for cross-team:** Deferred to Phase 4. In Phase 2, any agent can invoke any other agent's capability. This is acceptable for single-user, single-cluster deployments.

## 4. Phase 3: Production Security

### 4.1 Rootfs Image Specification

**Contents:**
- Linux kernel: 5.10+ minimal config (no modules, no filesystems beyond ext4, virtio drivers built-in)
- Init system: busybox init with a single init script
- hive-sidecar binary (statically compiled, placed at `/usr/local/bin/hive-sidecar`)
- busybox (provides sh, cat, ls, curl, wget, ping, ip, iptables, dnsmasq)
- CA certificates bundle
- `/workspace/` mount point (empty, populated via virtiofs)

**OpenClaw variant adds:**
- Node.js 20 LTS (minimal, no npm)
- OpenClaw dist directory

**Image size targets:** Base: < 50MB compressed. OpenClaw variant: < 150MB compressed.

**Build system:** The existing `rootfs/nixos/` Nix expressions build the rootfs. The GitHub Actions workflow runs `nix build` on both amd64 and arm64 runners, compresses with gzip, and uploads as release assets.

### 4.2 Network Policy Enforcement

Implemented in the rootfs init script (`/etc/init.d/S10-network`), configured by the sidecar at VM boot via environment variables passed through the Firecracker VM config.

| Policy | Implementation |
|---|---|
| `egress: none` | Firecracker VM config: no network device attached. Only vsock available. The sidecar communicates with NATS via vsock proxy (already implemented in `internal/sidecar/vsock_linux.go`). |
| `egress: restricted` | Firecracker VM config: tap device with host-side NAT. Inside VM: iptables rules DROP all outbound traffic. dnsmasq configured with allowlist domains. For each domain in `egress_allowlist`, a DNS entry resolves normally; all other queries return NXDOMAIN. iptables rules ACCEPT traffic to resolved IPs of allowlisted domains. |
| `egress: full` | Firecracker VM config: tap device with host-side NAT. No additional restrictions inside VM. |

**Data flow for restricted mode:**
1. hived reads `spec.network.egress_allowlist` from manifest
2. hived passes allowlist as JSON environment variable to the Firecracker VM config
3. VM boots, init script reads the variable
4. Init script writes dnsmasq config with allowlist entries
5. Init script writes iptables rules: default DROP on OUTPUT chain, ACCEPT for vsock, ACCEPT for DNS to localhost, ACCEPT for IPs resolved from allowlist

### 4.3 Seamless Tier Transition

**hived flag:** `--force-process-backend` (used by `hivectl dev`). When set, all agents use the process backend regardless of `spec.tier`. The reconciler passes this flag to the backend manager.

**Automatic inference (no flag):** If the node has KVM (`/dev/kvm` exists and is writable) and >= 4GB RAM, agents with `spec.tier: vm` use Firecracker. Otherwise, they fall back to process backend with a warning log.

**Environment parity guarantee:** The sidecar HTTP API, NATS subjects, capability routing, health heartbeats, and workspace paths are identical between backends. The only difference:
- Process backend: sidecar runs in-process (library mode), NATS via localhost TCP
- Firecracker backend: sidecar runs as standalone binary inside VM, NATS via vsock proxy

Agents must not detect or depend on which backend they're running on.

## 5. Phase 4: Hive Cloud

### 5.1 Architecture

Hive Cloud is a separate binary (`hive-cloud`) or a mode of `hived` (`hived --mode cloud`). It adds:
- HTTP API for deployment management (REST, not NATS -- external clients need HTTP)
- Tenant provisioning and lifecycle
- Autoscaler
- Billing/metering hooks

**Decision:** Separate binary (`hive-cloud`). It embeds `hived` functionality but adds the HTTP API layer, multi-tenancy, and autoscaler. The open-source `hived` remains single-tenant.

### 5.2 Deployment API

**Authentication:** Bearer token per tenant. Tokens issued during tenant provisioning. Passed in `Authorization: Bearer <token>` header.

**Endpoints:**

```
POST /api/v1/deployments
  Content-Type: multipart/form-data
  Body: tarball of cluster root directory
  -OR-
  Content-Type: application/json
  Body: { "git_url": "...", "git_ref": "main", "git_token": "..." }

  Response 201: { "deployment_id": "dep-xxx", "status": "validating" }
  Response 400: { "error": "validation failed", "details": [...] }

GET /api/v1/deployments/{id}
  Response 200: {
    "deployment_id": "dep-xxx",
    "status": "running" | "stopped" | "failed" | "validating" | "destroying",
    "agents": [
      { "id": "code-reviewer", "status": "running", "healthy": true, "uptime_seconds": 3600 }
    ],
    "created_at": "2026-03-10T12:00:00Z",
    "updated_at": "2026-03-10T13:00:00Z"
  }

DELETE /api/v1/deployments/{id}
  Response 202: { "deployment_id": "dep-xxx", "status": "destroying" }

POST /api/v1/deployments/{id}/trigger
  Content-Type: application/json
  Body: { "team": "ci-pipeline", "payload": { ... } }
  Response 202: { "run_id": "run-xxx", "status": "queued" }

GET /api/v1/deployments/{id}/runs/{run_id}
  Response 200: {
    "run_id": "run-xxx",
    "status": "running" | "completed" | "failed",
    "started_at": "...",
    "completed_at": "...",
    "result": { ... }  // pipeline output when completed
  }
```

### 5.3 Multi-Tenancy

**NATS isolation:** Use nats-server v2's multi-account feature. Each tenant gets a NATS account with its own subject namespace. Tenants cannot subscribe to or publish on other tenants' subjects. Account JWTs are generated during tenant provisioning and stored in the cloud control plane's database.

**State isolation:** Each tenant gets a separate SQLite database at `.state/tenants/{tenant_id}/state.db`. The existing `state.Store` is instantiated per-tenant.

**Compute isolation:** Each tenant's agents run in separate Firecracker VMs. The scheduler enforces per-tenant quotas:
- `max_concurrent_vms`: maximum VMs running simultaneously
- `max_total_memory_mb`: sum of all agent memory allocations
- `max_total_vcpus`: sum of all agent vCPU allocations

Quota exceeded returns HTTP 429 with `Retry-After` header.

### 5.4 Autoscaling

**Scaling unit:** Individual agent VM.

**Scale-up trigger:** NATS subject `hive.capabilities.{AGENT_ID}.{CAP}.request` has pending messages > 0 for > 5 seconds. Autoscaler starts additional agent replicas (up to `spec.resources.maxReplicas` if specified, default 1 = no scaling).

**Scale-to-zero trigger:** No capability invocations and no NATS messages for an agent for configurable idle timeout (default 10 minutes). Autoscaler stops the VM. State is persisted. Next invocation triggers a cold start.

**VM pool:** Maintain a pool of N pre-booted Firecracker VMs (N = configurable, default 5). VMs boot with base rootfs but no agent workspace. On deployment trigger, a pooled VM is assigned: workspace injected via virtiofs, sidecar configured, agent started. This reduces cold start from ~2s to ~200ms.

**Implementation:** New package `internal/autoscaler/`. Runs as a goroutine in `hive-cloud`. Reads NATS subject metrics via nats-server monitoring API. Makes scaling decisions on a 10-second loop. Calls the reconciler to adjust desired state.

### 5.5 GitHub Integration

GitHub Action (`brmurrell3/hive-action`):

**Inputs:**
- `cluster-root`: path to the .hive/ directory in the repo (required)
- `command`: `validate` | `deploy` | `trigger` (required)
- `hive-cloud-token`: authentication token (required for deploy/trigger)
- `hive-cloud-url`: API endpoint (default: `https://api.hive.dev`)
- `trigger-payload`: JSON payload for trigger command (optional)

**Behavior:**
1. Downloads `hivectl` binary from GitHub Releases (cached)
2. Runs `hivectl {command} --cluster-root {cluster-root} --target {hive-cloud-url}`
3. For `validate`: exits 0 on success, 1 on failure with error details in GitHub Actions summary
4. For `deploy`: pushes manifest bundle, waits for deployment to reach `running` status (timeout 120s)
5. For `trigger`: sends trigger, waits for run completion (timeout configurable)

### 5.6 Enterprise Self-Hosted Package

Single tarball: `hive-enterprise-{version}-{os}-{arch}.tar.gz`

Contents:
```
hive-enterprise/
  bin/
    hived
    hivectl
    hive-sidecar
    hive-agent
  images/
    hive-rootfs-amd64.ext4.gz
    hive-rootfs-arm64.ext4.gz
    hive-rootfs-openclaw-amd64.ext4.gz
  docs/
    getting-started.md
    operations.md
    ...
  LICENSE
```

**License validation:** hived reads a license file at `--license-file PATH` (or `HIVE_LICENSE_FILE` env var). The license is a signed JWT containing: tenant name, expiry date, max nodes, max agents. Signed with Hive's public key (embedded in the binary). Validation is offline -- no network call. Expired or invalid license: hived starts in read-only mode (existing agents run, no new deployments).

**Audit logging:** All control plane operations (agent start/stop, deployment create/delete, config changes) written to an append-only file at `{cluster-root}/.state/audit.log`. Format: JSON lines with timestamp, actor, action, resource, and outcome. File is append-only (O_APPEND | O_WRONLY). Not deletable while hived is running (file descriptor held open).

## 6. Deferred Decisions

These items are explicitly not decided in this document. They will be resolved during implementation when more context is available.

| Decision | When to resolve | Constraints |
|---|---|---|
| Agent callback protocol (HTTP vs stdin/stdout vs Unix socket) | Phase 1 implementation | Must work in both process and Firecracker backends |
| OpenClaw bridge: skill invocation vs chat message | Phase 2 implementation | Must support OpenClaw's existing skill format |
| Rootfs init system: busybox init vs custom Go init | Phase 3 implementation | Must support network policy setup and sidecar startup |
| hive-cloud: separate binary vs hived --mode cloud | Phase 4 implementation | Must keep open-source hived unaffected |
| Pricing granularity: agent-minute vs capability-invocation | Phase 4 launch | Must be measurable from existing metrics |
| NATS account provisioning: in-process vs external operator | Phase 4 implementation | Must support 100+ tenants on single cluster |
